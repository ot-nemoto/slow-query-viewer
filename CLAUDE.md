# CLAUDE.md

## 作業開始時のチェックリスト

1. `docs/product.md` を読みプロダクトを理解する
2. `docs/requirements.md` で今回のタスクに関連する要件を確認する
3. `docs/architecture.md` で実装方針を確認する
4. `docs/tasks.md` で実装順を確認し、タスクの状態は GitHub Issues で確認する

---

## ドキュメント構成ルール

実装・コード生成・DB設計を行う前に、以下のドキュメントが `docs/` 配下に揃っていることを確認すること。
不足しているドキュメントがある場合は、作業開始前にユーザーに確認または作成を促す。

### 必須ドキュメント

各ドキュメントは以下の必須セクションを含むこと。セクションが欠けている場合は作業前に追記する。

| ファイル | 役割 | 必須セクション |
|----------|------|--------------|
| `docs/product.md` | プロダクトの目的・対象ユーザー・ゴール | 目的、対象ユーザー、成功指標 |
| `docs/requirements.md` | 機能要件・非機能要件の一覧 | 機能要件（機能別）、非機能要件、画面一覧 |
| `docs/architecture.md` | 技術スタック・構成・実装方針 | 技術スタック、ディレクトリ構成、環境変数 |
| `docs/ui.md` | 画面一覧・遷移・コンポーネント・UI規約 | 画面一覧、画面遷移図（Mermaid）、画面機能仕様、各画面の表示状態（Loading / Empty / Error）、レイアウト構成、コンポーネント一覧、UI規約 |
| `docs/development.md` | 開発・運用手順 | ローカルセットアップ、環境変数、DB操作（該当する場合）、デプロイ手順 |
| `docs/testing.md` | 自動テスト方針・カバレッジ規約 | テスト種別、完了条件、カバレッジ方針、実行手順 |
| `docs/e2e-scenarios.md` | E2E テストシナリオ一覧 | 機能別テストシナリオ、確認観点 |
| `docs/tasks.md` | フェーズ構成と進捗管理 | フェーズ別マイルストーンリンク |

### 条件付き必須ドキュメント

以下は該当する実装・運用が存在する場合にのみ作成を必須とする。

| 条件 | ファイル | 役割 | 必須セクション |
|------|---------|------|--------------|
| 外部 REST API を提供する場合 | `docs/api.md` | 外部 REST API エンドポイント定義 | エンドポイント一覧、リクエスト・レスポンス定義、エラーレスポンス定義 |
| Server Actions を採用する場合 | `docs/actions.md` | Server Actions 定義（引数・戻り値・エラー） | Action 一覧、引数・戻り値・エラー定義、共通仕様 |
| 永続化 DB を利用する場合 | `docs/schema.md` | DB テーブル定義・リレーション・インデックス | テーブル定義（カラム・型・制約）、リレーション図、インデックス |
| 認証・認可フローがある場合 | `docs/auth.md` | 認証フロー・保護対象ルート・セッション管理 | 認証フロー、保護対象ルート、セッション管理 |
| 外部サービス連携がある場合 | `docs/integrations.md` | 外部サービス連携フロー・接続設定 | サービス一覧、連携フロー、環境変数 |
| デプロイ・インフラ構成が必要な場合 | `docs/infra.md` | インフラ構成・デプロイ手順 | 構成図、環境一覧、デプロイ手順 |

### README.md

`README.md` はリポジトリルートに配置し、GitHub のトップページに表示されるランディングページとして機能させる。

**記載する内容（他のドキュメントと重複させない）：**

| セクション | 内容 |
|-----------|------|
| プロダクト名・概要 | 一文でプロダクトの目的を説明 |
| 機能 | 主要機能の箇条書き（詳細は `docs/product.md` / `docs/requirements.md` に委譲） |
| ドキュメント | `docs/` 配下の全ドキュメントへのリンクテーブル |
| クイックスタート | 起動に最低限必要なコマンドのみ。詳細は `docs/development.md` へのリンクで委譲 |

## ドキュメント更新ルール

- 実装中に仕様変更が発生した場合、コード修正と同時に該当ドキュメントを更新する
- ドキュメントとコードの乖離が生じた場合は、原則ドキュメントを正として扱いコードを修正する。ただし意図的な仕様変更の場合はドキュメントを先に更新してからコードを変更する

## テストルール

テスト方針・カバレッジ規約・実行コマンドの詳細は [`docs/testing.md`](docs/testing.md) を参照。

### 完了条件

- **API ルートの実装はユニットテストの作成をもって完了とする**
- **`lib/` 配下のユーティリティ関数もユニットテストの作成をもって完了とする（パスはプロジェクト構成に従う）**
- UI コンポーネントのユニットテストは必須としない

## UI実装の動作確認ルール

- **UI を含む実装はアプリを起動した手動動作確認をもって完了とする**
- 実装完了後、コミット・プッシュの前に動作確認一覧を **Issue にコメントとして投稿** し、ユーザーに提示する
- ユーザーが動作確認を実施し、問題がなければコミット・プッシュに進む
- 動作確認中に一覧の追加・変更が発生した場合は、**新しいコメントを追加せず、最初に投稿したコメントを編集して更新する**

### 動作確認チェックリストの形式

以下の観点を網羅した表形式で投稿する：

| # | ユーザー | 手順 | 確認観点 | 期待値 |
|---|---|---|---|---|
| ... | ... | ... | ... | ... |

> 認証・ロール分離がないアプリでは「ユーザー」列は省略してよい。

### 最低限カバーすべき観点

- **認証・認可**: 未ログイン時のリダイレクト、ログイン成功・失敗
- **画面表示**: データが正しく表示されるか、空状態の表示
- **インタラクション**: ボタン・フォームの操作、状態変化（選択・入力）
- **保存・更新**: 保存後のフィードバック表示、リロード後のデータ永続化
- **ユーザー分離**: 複数ユーザーでデータが混在しないか（該当する場合）

## バージョン準拠ルール

- フレームワーク・ライブラリのバージョンは **常に `package.json` を確認して**、そのバージョンの仕様に従う
- 古いバージョンの知識や一般的な慣習を根拠にしない
- 外部ツール（Copilot 等）やレビュアーからフレームワーク・ライブラリの慣習に関する指摘を受けた場合、`package.json` のバージョンに基づく仕様を確認してから採否を判断する
- バージョン固有の仕様・既知の指摘パターンは `docs/architecture.md` に記載する

## コーディングルール

- コメント: 自明でないロジックにのみ付ける（過剰なコメント不要）
- エラーハンドリング: 外部入力・API レスポンスの境界でのみ行う
- 言語・ツール・フォーマッタなどプロジェクト固有の選択は `docs/architecture.md` を参照

## GitHub Issues 管理ルール

新規タスク・不具合はすべて GitHub Issues で管理する。

### ブランチ・マージルール

- `develop` および `master` へのマージは **必ず PR を作成**し、ユーザーの承認を得てからマージする
- 直接マージ（`git merge`）は行わない

### Issue 作成ルール

| 項目 | ルール |
|------|--------|
| タイトル | `[T番号] タスク名` の形式（例: `[T60] XXX機能追加`） |
| T番号 | 直前の最大番号 + 1 で採番する |
| Milestone | 対象フェーズの Milestone をアサインする |
| Label | `enhancement`（新機能）/ `fix`（不具合）/ `testing`（テスト）/ `documentation`（ドキュメント）/ `ci`（CI/CD・GitHub Actions・Dependabot などインフラ寄りの変更）から選ぶ |
| 本文 | 概要・要件・依存 Issue を記載する |

### Issue クローズルール

Issue をクローズする際は、**必ず関連するコミット ID をコメントに含める**。

```bash
gh issue close <番号> --comment "完了。実装コミット: <short-hash> (<コミットメッセージ>)

<補足説明（任意）>"
```

## バージョニングルール

`package.json` の `version` はセマンティックバージョニング（SemVer）に従う。

### バンプ方法

> **バージョンバンプは `bump-version.yml` ワークフローが自動で行う。**
> master への PR にラベルを付けるだけでよく、`package.json` の手動更新は不要。

| ラベル | バージョン操作 | 例 |
|--------|-------------|-----|
| `bump:minor` | マイナー +1、パッチ → 0 | `0.1.0` → `0.2.0` |
| `bump:patch` | パッチ +1 | `0.2.0` → `0.2.1` |

### バンプの判断基準

| 変更の種類 | 使用ラベル |
|-----------|-----------|
| 新機能追加（フェーズ完了など） | `bump:minor` |
| バグ修正・小改善のみ | `bump:patch` |

## Git 操作ルール

- `git commit` / `git push` はユーザーの **明示的な承認（「OK」など）を受けてから** 実行する
- 実装・ドキュメント更新が完了したら、変更内容をユーザーに提示して承認を待つ
- 承認前にコミット・プッシュを自動で行わない

### Issue 着手フロー

Issue に着手する際は、以下の順で進める：

1. Issue の仕様・現状コードを確認し、作業タスクを整理する
2. **Issue にコメントとして**作業タスク（チェックリスト形式）を投稿する
3. ユーザーに確認・承認を得る
4. `feature/t<番号>-<概要>` の形式でブランチを切る
5. 承認後に実装を開始する

> Issue 本文は「何をするか（仕様）」、コメントは「どうやるか（作業計画・進捗）」として使い分ける。

### Issue 単位の作業フロー

1 つの Issue に対応する実装が完了したら、以下の順で処理する：

1. 変更内容をユーザーに提示して承認を待つ
2. 承認後、`git commit` → `git push` を実行する
3. PR を作成し、ユーザーのレビューを待つ（マージは自分でしない）
4. マージ後、対応する GitHub Issue を関連コミット ID を添えてクローズする

### 作業スコープルール

- **指示されたタスク（Issue）のみを実行する**。ユーザーが明示的に指定していないタスクは、自己判断で着手しない
- 複数 Issue をまとめて依頼された場合も、1 Issue ごとに完了・承認を経てから次に進む
- 「マイルストーン X を進める」など範囲指定の指示を受けた場合は、着手前に対象 Issue の一覧と実施順をユーザーに提示して確認をとる

## PR レビュー指摘の対応ルール

- Copilot・レビュアーからの指摘は、マージ前に同一ブランチ内で対応する
- 対応不要と判断した場合はその理由をコメントで返す
- 指摘対応のコミットメッセージは `fix: <内容>` とする
- 指摘に対応したら、該当コメントに対応済みの旨をコミット ID を添えて返信し、スレッドを Resolve する

```
対応済み。修正コミット: <short-hash> (<コミットメッセージ>)

<補足説明（任意）>
```
